## 一、项目是什么

**GNG 管理系统**是一个面向货柜/物流业务的 Web 应用，核心能力包括：

1. **Google Sheet 模块**：在线表格化管理柜号订单（23+ 业务字段）
2. **订单管理**：简化版订单 CRUD + 一键 Gmail 检索解析
3. **解析链路**：Gmail 搜邮件 → 下载 Excel/CSV 附件 → 解析派送明细 → 入库
4. **解析结果 / 日志 / 仓库汇总**：按柜号 + 批次号追踪每次解析

技术栈：**Next.js 14 (App Router)** + **TypeScript** + **Prisma 5** + **PostgreSQL** + **Tailwind CSS** + **Gmail API (googleapis)** + **ExcelJS**

**约定：**

- 页面组件：`"use client"`，负责 UI 与 `fetch`
- API 路由：Server 端，负责鉴权、校验、调 Service、返回 `success()` / `error()`
- 复杂业务：放在 `src/lib/*-service.ts`，不在 route 里堆逻辑
- 所有返回前：`serialize(data)` 处理 BigInt

## 五、认证与权限

### 5.1 双 Token + Cookie

| Cookie 名 | 用途 | 默认有效期 |
|-----------|------|------------|
| `gng_access_token` | API/页面访问 | 15 分钟（`JWT_ACCESS_EXPIRES`） |
| `gng_refresh_token` | 刷新（当前 middleware 仅检查 access） | 7 天 |

登录流程：

```
POST /api/v1/auth/login  { email, password }
  → bcrypt 校验 users 表
  → signAccessToken + signRefreshToken
  → Set-Cookie
```

### 5.2 两层守卫

1. **`middleware.ts`**（Edge）：无 Token → 页面跳 `/login`，API 返回 401  
2. **`requireUser(request)`**（API 内）：解析 JWT payload 得到 `{ id, role, ... }`

### 5.3 角色

| 角色 | 写/删 | 说明 |
|------|-------|------|
| `admin` | ✓ | 全部权限 |
| `operator` | ✓ | 业务操作 |
| `viewer` | ✗ | 只读 |

前端通过 `/api/v1/auth/me` 获取角色，控制按钮显示；后端仍必须用 `canWrite()` 二次校验。

## 六、数据模型（Prisma）

### 6.1 两条业务线

**线 A：Google Sheet（大表，软删除）**

```
users ──→ google_sheet ──→ google_sheet_history
              │
              └── container_parse_meta（按柜号存 Gmail 元数据）
```

**线 B：订单 + Gmail 解析（批次制）**

```
orders ──→ containers（每次解析一条，含 batch_no）
              │
              ├── attachments（每个 Excel/CSV 附件）
              │       └── delivery_items（解析明细行）
              ├── parse_logs（步骤日志）
              └── warehouse_summaries（按仓库汇总，含 batch_no）
```

### 6.2 关键表说明

| 表 | 说明 |
|----|------|
| `orders` | 订单管理主表，柜号、客户、日期等 |
| `containers` | 一次 Gmail 解析 = 一行，`batch_no` 区分同柜多次解析 |
| `attachments` | 邮件内每个可解析附件 |
| `delivery_items` | 附件内每一行派送明细（FBA、仓库、箱数…） |
| `parse_logs` | `step` + `status` + `message` |
| `warehouse_summaries` | 按 `container_no + warehouse_code + batch_no` 汇总 |
| `google_sheet` | 23 字段业务大表，柜号唯一 |

### 6.3 枚举

- `operation_type`：`fcl`（整柜）/ `lcl`（拆柜）— 仅 google_sheet 使用  
- `log_status`：`success` / `failed` / `warning`  
- `user_role`：`admin` / `operator` / `viewer`

## 七、核心业务：订单 Gmail 解析

**入口：** 订单页点击「检索」→ `POST /api/v1/orders/[id]/search`

**核心文件：** `src/lib/order-parse-service.ts`

### 流程图

```
1. 读取 orders 记录
2. 生成 batch_no = 柜号-yyyyMMddHHmmss
3. 创建 containers 行（parse_status=parsing）
4. Gmail 搜索
   ├─ 先: from:{GMAIL_DEFAULT_SENDER} {柜号}
   └─ 若无: 仅 {柜号}
5. pickBestEmailForParse → getEmailDetail
6. 筛选 isParseable 附件（.xlsx / .csv）
7. 对每个附件：
   ├─ 下载（事务外）
   ├─ parseDeliveryFileBuffer
   ├─ 写入 attachments + delivery_items + warehouse_summaries
   └─ 记录 parse_logs
8. 更新 containers 邮件字段与 parse_status
```

**设计要点：**

- 网络 IO 与 Excel 解析 **不在** Prisma 事务内，避免 `25P02` 事务 aborted  
- 失败时错误写入 `error_message` 和 `parse_logs`，前端 toast 提示

## 八、Gmail 集成

### 8.1 环境变量

```env
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
GOOGLE_REDIRECT_URI=http://localhost:3000/api/v1/gmail/callback
```

### 8.2 OAuth 流程

```
GET /api/v1/gmail/auth        → 跳转 Google 授权
GET /api/v1/gmail/callback    → 换 Token，写入 gmail_* Cookie
```

Token 由 `gmail-tokens.ts` 管理，解析 API 通过 `resolveGmailTokens(request)` 获取。

### 8.3 关键函数（`gmail.ts`）

| 函数 | 作用 |
|------|------|
| `searchEmailsByContainer` | 按柜号搜邮件，支持发件人 fallback |
| `getEmailDetail` | 标题、正文、附件列表 |
| `downloadAttachmentBuffer` | 下载附件二进制 |
| `pickBestEmailForParse` | 多封邮件时选最优（优先有 Excel/CSV） |

附件识别：`isParseable = isExcel || isCsv`（支持 `表格_xxx.csv`）

## 九、Excel/CSV 解析

**文件：** `delivery-excel-parser.ts` + `excel-utils.ts`

1. `parseDeliveryFileBuffer(buffer, filename, containerNo)`  
   - `.csv` → 手工解析 CSV 为二维矩阵  
   - 否则 → ExcelJS 读 xlsx  
2. `detectHeaderRow` → 找表头行（含 FBA、仓库、箱数等关键词）  
3. `FIELD_ALIASES` → 列名映射到标准字段  
4. `aggregateWarehouseSummaries` → 按仓库汇总箱数  

解析出的字段：`customer_code`, `fba_id`, `reference_id`, `cbm`, `weight`, `carton_count`, `warehouse_code`, `delivery_method` 等。

## 十三、校验与映射模式

项目统一用 **Zod** 做入参校验：

```typescript
// 典型 API POST 模式
const parsed = orderCreateSchema.safeParse(body);
if (!parsed.success) {
  return error("Validation failed", 400, parsed.error.issues.map(...));
}
const created = await prisma.orders.create({
  data: buildOrderCreateInput(parsed.data, BigInt(user.id)),
});
return success(serialize(created));
```

- **Schema 文件**：`*-validators.ts`  
- **Mapper 文件**：`*-mapper.ts`（DTO → Prisma `CreateInput` / `UpdateInput`）  
- **List Query 文件**：`*-list-query.ts`（搜索 where、排序 orderBy）

新增模块时复制这套三板斧即可。

## 十九、常见问题

**Q: BigInt 序列化报错？**  
A: API 返回前必须 `serialize(data)`。

**Q: middleware 里能用 Prisma 吗？**  
A: 不能，Edge Runtime 不支持；只做 Cookie 检查。

**Q: `container-parse-service` 和 `order-parse-service` 区别？**  
A: 前者服务 Google Sheet / by-no 旧 API；后者是订单管理的主解析链路（含 attachments 批次模型）。

**Q: 为什么有两套 containers 概念？**  
A: Google Sheet 按柜号唯一；订单解析按「每次检索」多行 + `batch_no`。Prisma 中 `containers` 表已统一为后者。
