## 一、项目背景与目标

### 1.1 业务背景

当前公司使用 **Google Sheet** 管理集装箱操作记录，存在以下痛点：

- ❌ 数据安全性差（多人协作易出错）
- ❌ 无法实现权限控制
- ❌ 缺乏自动化处理能力（邮件解析、Excel导入）
- ❌ 移动端体验不佳
- ❌ 无法与其他系统集成

### 1.2 项目目标

构建一个 **Web 版集装箱操作管理系统**，实现：

✅ **核心目标**
- 完全替代 Google Sheet 工作流
- 提供企业级数据安全和权限管理
- 支持邮件自动解析和 Excel 智能导入
- 实现 Google Sheet 级别的表格交互体验

✅ **用户体验目标**
- 操作流畅，响应迅速（< 100ms）
- 支持桌面端和移动端
- 提供实时数据保存和撤销功能

✅ **技术目标**
- 采用现代化技术栈（Next.js + TypeScript）
- 代码结构清晰，易于维护
- 支持 Docker 容器化部署

### 1.3 目标用户

| 用户角色 | 权限范围 | 主要操作 |
|---------|---------|---------|
| **管理员** | 全部权限 | 数据管理、系统配置、用户管理 |
| **操作员** | 数据 CRUD | 录入/编辑/删除集装箱记录 |
| **查看者** | 只读权限 | 查看数据、导出报表 |

## 三、功能需求详解

### 3.1 模块一：用户认证与权限管理

#### 功能描述
实现安全的用户登录系统，确保数据访问安全。

#### 详细需求

##### 3.1.1 用户注册与登录
- **登录方式**：邮箱 + 密码
- **可选方式**：验证码登录、Google OAuth、GitHub OAuth
- **密码规则**：至少 8 位，包含大小写字母和数字
- **会话管理**：支持"记住我"功能（7天有效期）

##### 3.1.2 权限控制
- **未登录状态**：
  - 重定向到登录页面
  - 无法访问任何 API 接口
  
- **已登录状态**：
  - API 请求自动携带认证 Token
  - 根据角色显示不同功能按钮
  - 数据隔离：用户只能看到自己有权访问的数据

##### 3.1.3 安全要求
- ✅ 密码加密存储（bcrypt，salt rounds ≥ 12）
- ✅ Token 有效期：Access Token 15分钟，Refresh Token 7天
- ✅ 支持 CSRF 防护
- ✅ 登录失败次数限制（5次/15分钟）

#### 验收标准
- [ ] 未登录用户访问任意页面 → 重定向到 `/login`
- [ ] 登录后 Token 自动附加到请求头
- [ ] 不同角色看到不同的 UI 元素
- [ ] Token 过期后自动刷新或提示重新登录

### 3.3 模块三：Google Sheet 风格表格交互

#### 功能描述
提供类似 Google Sheet 的流畅交互体验。

#### 详细需求

##### 3.3.1 单元格编辑
- **触发方式**：单击选中，双击/Enter 进入编辑模式
- **编辑体验**：
  - 光标定位准确
  - 支持键盘导航（↑↓←→ Tab Enter Escape）
  - 文本超出时显示省略号，hover 显示完整内容

##### 3.3.2 行操作
- **新增行**：
  - 在表格顶部插入新行
  - 或在指定位置插入（右键菜单）
  - 快捷键：Ctrl+Enter（在当前行下方插入）
  
- **删除行**：
  - 单行删除：行末尾显示删除按钮（hover 时出现）
  - 批量删除：勾选复选框 → 点击工具栏"删除"按钮

##### 3.3.3 排序功能
- **单列排序**：点击列头切换 升序 → 降序 → 默认
- **多列排序**：按住 Shift 点击多列头（优先级从左到右）
- **排序图标**：升序 ↑，降序 ↓，默认无图标
- **支持数据类型**：文本、数字、日期

##### 3.3.4 筛选功能
- **筛选入口**：每列头部显示筛选图标
- **筛选项类型**：
  - 文本列：搜索框 + 多选列表

  - 日期列：日期选择器（开始/结束）
  - 布尔列：是/否/全部
- **组合筛选**：多列筛选条件 AND 关系
- **清除筛选**：一键清除所有筛选条件

##### 3.3.5 搜索功能
- **全局搜索**：页面顶部搜索框
- **搜索范围**：柜号、MBL、客户、司机等关键字段
- **搜索逻辑**：
  - 支持模糊匹配（包含即可）
  - 支持精确匹配（加引号 `"EITU1829176"`）
  - 实时搜索（输入 300ms 后执行）

##### 3.3.6 列宽与顺序调整
- **列宽调整**：
  - 鼠标拖动列边界线

  - 拖拽列头改变位置

  - 支持撤销
  
- **删除空行**：
  - 定义"空行"标准：所有可选字段都为空的行
  - 批量删除并显示删除数量
  - 操作前二次确认

- [ ] 表格操作流畅，无明显卡顿（1000 行以内 < 16ms 响应）
- [ ] 所有快捷键正常工作

### 3.4 模块四：业务逻辑功能

#### 功能描述
实现集装箱业务相关的智能提醒和数据校验。

#### 详细需求

##### 3.4.1 LFD 到期颜色提醒
- **预警规则**：
  - 🔴 **红色**：今天到期或已过期（LFD ≤ 今天）
  - 🟡 **黄色**：3天内到期（今天 < LFD ≤ 今天+3天）
  - 🟢 **绿色**：超过3天（LFD > 今天+3天）
  - ⚪ **灰色**：未填写 LFD

- **展示位置**：
  - LFD 列的单元格背景色
  - 整行淡色背景（优先级取最高）
  
- **交互细节**：
  - Hover 显示"距离到期 X 天"
  - 红色行闪烁提示（可选，可关闭）

##### 3.4.2 柜号格式校验
- **格式规则**：4个大写字母 + 7个数字
- **正则表达式**：`/^[A-Z]{4}\d{7}$/`
- **示例**：
  - ✅ 合法：`EITU1829176`, `EMCU1603808`, `TIIU4620869`
  - ❌ 非法：`eitu1829176`, `EITU182917`, `EITU18291767`

- **校验时机**：
  - 输入时实时校验（blur 事件）
  - 校验失败显示红色边框 + 错误提示
  - 保存时强制校验（不通过则阻止保存）

##### 3.4.3 多维度筛选器
**快速筛选面板**（侧边栏或顶部栏）：

| 筛选项 | 类型 | 可选值示例 |
|-------|------|-----------|
| 客户 | 多选下拉 | MEITONG, KJY, HEXING... |
| 码头 | 多选下拉 | ETS, OICT |

### 3.5 模块五：邮件自动解析系统

#### 功能描述
自动从 Gmail 获取集装箱相关邮件，解析 Excel 附件，提取数据并存入数据库。

#### 详细需求

##### 3.5.1 Gmail 邮件获取
- **搜索条件**：
  ```
  主题包含柜号（如 "EGSU6027772"）
  发件人为指定邮箱（如 *@ggtransport.in）
  有附件（.xlsx/.xls/.csv）
  时间范围：最近 30 天
  ```

- **API 使用**：Google Gmail API
  - 认证方式：OAuth 2.0（Service Account 或 User Token）
  - 权限范围：`gmail.readonly` 或 `gmail.modify`
  - 速率限制：每秒 250 次请求（配额）

- **邮件选择策略**：
  - 优先选择最新邮件
  - 如果找到多封邮件，选择附件最大的
  - 记录所有候选邮件到 parse_logs

##### 3.5.2 Excel 附件下载与解析
- **支持的格式**：
  - `.xlsx`（Excel 2007+）
  - `.xls`（Excel 97-2003）
  - `.csv`（逗号分隔值）

- **解析流程**：
  1. 下载附件到临时目录
  2. 使用 `xlsx` 库读取工作簿
  3. 自动检测表头行（查找包含"FBA ID"、"Reference"、"仓库"等关键词的行）
  4. 提取数据行（从表头下一行开始，直到遇到空行）
  5. 字段映射：将 Excel 列名映射到数据库字段

- **表头映射示例**：

  | Excel 列名 | 数据库字段 | 数据类型 |
  |-----------|-----------|---------|
  | FBA ID / FBA | fba_id | TEXT |
  | Reference ID / PO | reference_id | TEXT |
  | CBM / 体积 | cbm | NUMERIC(10,2) |
  | Weight / 重量(kg) | weight | NUMERIC(10,2) |
  | Cartons / 箱数 | carton_count | INTEGER |
  | Warehouse Code / 仓库 | warehouse_code | VARCHAR(50) |
  | Delivery Method / 派送方式 | delivery_method | VARCHAR(50) |
  | Customer Note / 客户备注 | customer_note | TEXT |

- **异常处理**：
  - 表头缺失：尝试模糊匹配列名
  - 数据类型错误：标记为 warning，跳过该字段
  - 空行：忽略
  - 重复行：保留第一条，后续标记 duplicate

##### 3.5.3 数据入库与汇总
- **入库流程**（事务保证）：
  1. 清除该柜号的旧明细数据（如果重新解析）
  2. 插入新的 delivery_items 记录
  3. 更新 containers 表的解析状态和信息
  4. 计算 warehouse_summaries 汇总数据
  5. 写入 parse_logs 日志

- **仓库汇总计算逻辑**：
  ```sql
  INSERT INTO warehouse_summaries (container_no, warehouse_code, total_cartons, item_count)
  SELECT 
    container_no,
    warehouse_code,
    SUM(carton_count) AS total_cartons,
    COUNT(*) AS item_count
  FROM delivery_items
  WHERE container_no = $1
  GROUP BY container_no, warehouse_code
  ON CONFLICT (container_no, warehouse_code)
  DO UPDATE SET
    total_cartons = EXCLUDED.total_cartons,
    item_count = EXCLUDED.item_count,
    updated_at = NOW();
  ```

##### 3.5.4 解析状态机
```
pending → parsing → success
                ↘ failed (可重试)
                ↘ partial_success (有 warning)
```

- **状态说明**：
  - `pending`：待解析（刚创建柜号记录）
  - `parsing`：正在解析中
  - `success`：解析成功，无错误
  - `failed`：解析失败（致命错误）
  - `partial_success`：部分成功（有 warning）

#### 触发方式
- **手动触发**：用户在详情页点击"重新解析邮件"
- **定时任务**（加分项）：每小时检查一次是否有新邮件
- **批量触发**（加分项）：管理员可选择多个柜号批量解析

#### 验收标准
- [ ] 能正确搜索并下载 Gmail 附件
- [ ] 解析成功率 > 95%（针对标准格式 Excel）
- [ ] 异常情况有明确的日志记录
- [ ] 重复解析能正确覆盖旧数据
- [ ] 汇总数据计算准确
