## 1. 项目概览

本系统是一个**集装箱操作管理系统**，用于管理海运集装箱的进出口操作流程。

### 核心功能

| 功能 | 说明 |
|------|------|
| 用户认证 | 邮箱密码登录、JWT Token 认证、基于角色的权限控制 |
| 集装箱管理 | 增删改查、模糊搜索、列排序、行拖拽排序、历史版本 |
| 客户管理 | 单表 CRUD 示例（教学用） |

### 用户角色

| 角色 | 权限 |
|------|------|
| admin | 全部权限（增删改查 + 用户管理） |
| operator | 写权限（增删改） |
| viewer | 只读权限（只能查看） |

### 后端

| 技术 | 版本 | 用途 |
|------|------|------|
| **Next.js API Routes** | - | 服务端 API（App Router 模式） |
| **Prisma** | 5.x | TypeScript ORM，类型安全的数据库访问 |
| **PostgreSQL** | - | 关系型数据库 |
| **jose** | 6.x | JWT 签发与验证（Edge Runtime 兼容） |
| **bcryptjs** | - | 密码哈希加密 |

### 为什么选择这些技术？

```
Q: 为什么用 Next.js 而不是纯 React？
A: Next.js 提供了全栈能力：
   - API Routes → 不需要单独的 Express 后端
   - App Router → 文件系统路由，自动生成页面
   - Middleware → 路由守卫，不需要额外配置

Q: 为什么用 Prisma 而不是原生 SQL？
A: Prisma 提供了类型安全的数据库访问：
   - 自动生成 TypeScript 类型
   - 查询语法直观（prisma.users.findMany()）
   - 支持事务、关联查询、迁移

Q: 为什么用 jose 而不是 jsonwebtoken？
A: Next.js Middleware 运行在 Edge Runtime：
   - jsonwebtoken 依赖 Node.js 的 crypto 模块，Edge 不支持
   - jose 使用 Web Crypto API，Edge Runtime 兼容

Q: 为什么用 zod 而不是手写验证？
A: zod 提供声明式验证：
   - 一行代码定义规则：z.string().min(8, "密码至少8位")
   - 自动生成 TypeScript 类型
   - 错误信息标准化
```

## 5. 核心架构

### 5.1 请求生命周期

```
用户访问页面
    ↓
Next.js Middleware（src/middleware.ts）
    ├── 未登录 → 重定向到 /login
    └── 已登录 → 继续
    ↓
Page Component（src/app/containers/page.tsx）
    ├── 加载用户信息 → GET /api/v1/auth/me
    └── 加载列表数据 → GET /api/v1/containers
    ↓
API Route Handler（src/app/api/v1/containers/route.ts）
    ├── requireUser() → 验证 JWT Token
    ├── 权限检查 → canWrite(user.role)
    ├── 数据验证 → zodSchema.safeParse(body)
    ├── 数据库操作 → prisma.xxx.findMany() / create() / update()
    └── 返回响应 → success(data) / error(message)
    ↓
前端更新状态 → useState / useEffect
    ↓
页面重新渲染
```

### 5.2 API 响应格式

所有 API 都遵循统一的响应格式：

```typescript
// 成功响应
{
  "code": 200,
  "message": "success",
  "data": { ... },                    // 数据
  "pagination": {                      // 分页信息（列表接口才有）
    "page": 1,
    "pageSize": 50,
    "total": 100,
    "totalPages": 2
  }
}

// 错误响应
{
  "code": 400,
  "message": "验证失败",
  "errors": [                          // 详细错误（可选）
    { "field": "container_no", "message": "柜号格式：4位大写字母+7位数字" }
  ]
}
```

### 5.3 认证流程

```
┌─────────────────────────────────────────────────┐
│                   登录流程                        │
│                                                   │
│  用户输入邮箱密码                                  │
│       ↓                                           │
│  POST /api/v1/auth/login                          │
│       ↓                                           │
│  bcrypt.compare(密码, 数据库哈希)                   │
│       ↓ 验证通过                                   │
│  signAccessToken(user) → 短期 JWT (15分钟)        │
│  signRefreshToken(user) → 长期 JWT (7天)          │
│       ↓                                           │
│  setAuthCookies() → 写入 HttpOnly Cookie          │
│       ↓                                           │
│  前端 router.push("/containers")                   │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│                 请求鉴权流程                       │
│                                                   │
│  浏览器发送请求（自动携带 Cookie）                   │
│       ↓                                           │
│  Middleware 检查 Cookie 中有无 Access Token         │
│       ├── 无 → 401 或重定向到 /login               │
│       └── 有 → 继续                                │
│       ↓                                           │
│  API Route 中 requireUser(request)                 │
│       ↓                                           │
│  verifyToken(token, "access") → 解析 JWT           │
│       ├── 无效 → 返回 null → 401                   │
│       └── 有效 → 返回 AuthUser 信息                │
│       ↓                                           │
│  使用 user.id / user.role 进行业务操作              │
└─────────────────────────────────────────────────┘
```

### 6.2 软删除设计

所有业务表都采用**软删除**模式：

```sql
-- 删除时不执行 DELETE，而是设置 deleted_at
UPDATE containers SET deleted_at = NOW() WHERE id = 1;

-- 查询时过滤已删除的记录
SELECT * FROM containers WHERE deleted_at IS NULL;
```

在 Prisma 中：

```typescript
// 查询时添加条件
const items = await prisma.containers.findMany({
  where: { deleted_at: null },  // 只查未删除的
});

// "删除"操作实际是更新
await prisma.containers.update({
  where: { id },
  data: { deleted_at: new Date() },
});
```

### 6.3 BigInt 序列化问题

PostgreSQL 的 `BIGINT` 在 Prisma 中映射为 JavaScript `BigInt`，但 `JSON.stringify` 不支持 BigInt。

```typescript
// ❌ 直接返回会报错
return NextResponse.json({ id: BigInt(123) });
// TypeError: Do not know how to serialize a BigInt

// ✅ 使用 serialize() 转换
import { serialize } from "@/lib/serialize";
return NextResponse.json(serialize({ id: BigInt(123) }));
// { "id": "123" }  → BigInt 变成字符串
```

## 8. 单表 CRUD 教程

> 以"客户管理"为例，从零开始实现一个完整的单表增删改查。

### 8.1 第一步：定义数据库模型

在 `prisma/schema.prisma` 中添加模型：

```prisma
model customers {
  id          BigInt   @id @default(autoincrement())  // 自增主键
  name        String   @db.VarChar(100)               // 客户名称（必填）
  contact     String?  @db.VarChar(50)                // 联系人（可选）
  phone       String?  @db.VarChar(20)                // 电话（可选）
  email       String?  @db.VarChar(100)               // 邮箱（可选）
  address     String?  @db.VarChar(200)               // 地址（可选）
  is_active   Boolean  @default(true)                 // 是否启用
  remarks     String?                                 // 备注
  created_by  BigInt                                  // 创建人（外键）
  updated_by  BigInt?                                // 更新人（外键）
  created_at  DateTime @default(now()) @db.Timestamptz(6)  // 创建时间
  updated_at  DateTime @default(now()) @db.Timestamptz(6)  // 更新时间
  deleted_at  DateTime? @db.Timestamptz(6)            // 软删除时间

  // 关联关系
  creator users? @relation("customers_created_by", fields: [created_by], references: [id])
  updater users? @relation("customers_updated_by", fields: [updated_by], references: [id])

  @@index([name], map: "idx_customers_name")
  @@index([created_at(sort: Desc)], map: "idx_customers_created_at")
}
```

然后在 `users` 模型中添加反向关联：

```prisma
model users {
  // ... 已有字段 ...
  customers_created  customers[] @relation("customers_created_by")
  customers_updated  customers[] @relation("customers_updated_by")
}
```

运行命令：

```bash
npx prisma generate   # 生成 TypeScript 类型
npx prisma db push    # 同步到数据库（开发环境）
```

### 8.2 第二步：创建 API 路由

#### 列表 + 创建：`src/app/api/v1/customers/route.ts`

```typescript
import { prisma } from "@/lib/prisma";
import { error, success } from "@/lib/api-response";
import { requireUser } from "@/lib/require-user";
import { serialize } from "@/lib/serialize";
import { canWrite } from "@/lib/auth";
import { z } from "zod";

// 验证规则
const customerSchema = z.object({
  name: z.string().min(1, "客户名称不能为空").max(100),
  contact: z.string().max(50).optional().nullable(),
  phone: z.string().max(20).optional().nullable(),
  // ...其他字段
});

// GET /api/v1/customers - 列表查询
export async function GET(request: Request) {
  // 1. 鉴权
  const user = await requireUser(request as import("next/server").NextRequest);
  if (!user) return error("Unauthorized", 401);

  // 2. 解析查询参数
  const { searchParams } = new URL(request.url);
  const page = Math.max(1, Number(searchParams.get("page") ?? 1));
  const pageSize = Math.min(200, Math.max(1, Number(searchParams.get("pageSize") ?? 20)));
  const search = searchParams.get("search")?.trim();

  // 3. 构建查询条件（软删除过滤 + 搜索）
  const where = {
    deleted_at: null,
    ...(search ? { OR: [
      { name: { contains: search, mode: "insensitive" } },
    ] } : {}),
  };

  // 4. 并行查询总数和列表
  const [total, items] = await Promise.all([
    prisma.customers.count({ where }),
    prisma.customers.findMany({
      where,
      orderBy: { created_at: "desc" },
      skip: (page - 1) * pageSize,
      take: pageSize,
    }),
  ]);

  // 5. 返回分页数据（注意 serialize 处理 BigInt）
  return success(serialize(items), {
    page, pageSize, total,
    totalPages: Math.ceil(total / pageSize),
  });
}

// POST /api/v1/customers - 创建
export async function POST(request: Request) {
  const user = await requireUser(request as import("next/server").NextRequest);
  if (!user) return error("Unauthorized", 401);
  if (!canWrite(user.role)) return error("权限不足", 403);

  const body = await request.json();
  const parsed = customerSchema.safeParse(body);
  if (!parsed.success) return error("验证失败", 400);

  const created = await prisma.customers.create({
    data: {
      ...parsed.data,
      created_by: BigInt(user.id),
      updated_by: BigInt(user.id),
    },
  });

  return success(serialize(created));
}
```

#### 详情 + 更新 + 删除：`src/app/api/v1/customers/[id]/route.ts`

```typescript
// GET    /api/v1/customers/[id] → 获取详情
// PUT    /api/v1/customers/[id] → 更新
// DELETE /api/v1/customers/[id] → 软删除

// 关键代码片段：

// 动态路由参数获取
type Params = { params: { id: string } };
export async function GET(request: Request, { params }: Params) {
  const id = BigInt(params.id);  // 字符串转 BigInt
  const item = await prisma.customers.findFirst({
    where: { id, deleted_at: null },
  });
  if (!item) return error("客户不存在", 404);
  return success(serialize(item));
}

// 软删除
export async function DELETE(request: Request, { params }: Params) {
  await prisma.customers.update({
    where: { id: BigInt(params.id) },
    data: { deleted_at: new Date() },
  });
  return success({ id: params.id });
}
```

## 10. 常见问题与调试

### Q1: `BigInt` 序列化报错

```
TypeError: Do not know how to serialize a BigInt
```

**解决**：所有 API 返回前调用 `serialize(data)`

### Q2: Hydration 错误 `<div> cannot be a child of <tbody>`

**原因**：`@dnd-kit` 的 `DndContext` 会注入辅助 `<div>`

**解决**：将 `DndContext` 放在 `<table>` 外层

### Q3: `sensors` 数组大小变化警告

```
Warning: The final argument passed to useEffect changed size between renders.
```

**原因**：`sensors={condition ? sensors : []}` 导致数组大小变化

**解决**：始终传入 `sensors={sensors}`，通过 `disabled` 控制行为

### Q4: `prisma db push` vs `prisma migrate`

| 命令 | 适用场景 | 说明 |
|------|---------|------|
| `prisma db push` | 开发环境 | 直接同步 schema 到数据库，不生成迁移文件 |
| `prisma migrate dev` | 团队开发 | 生成迁移 SQL 文件，可版本控制 |
| `prisma migrate deploy` | 生产环境 | 执行已有迁移文件 |
